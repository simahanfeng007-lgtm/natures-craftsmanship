# 项目收口质检

## 工具

- `scan_project`
- `run_python_quality_check`
- `run_python_tests`
- `diagnose_project`
- `evaluate_quality_gate`
- `runtime_tool_alignment_check`
- `build_delivery_standardization`
- `return_analysis`

## 1. 定位
项目收口质检是交付前最后一道质量门。不替代代码诊断、文件管理、终端执行、项目打包，而是把它们的检查结果统一裁决为：通过 / 部分通过 / 不通过 / 禁止发布。覆盖项目结构、语法测试、关键路径smoke、工具对齐、打包完整性、运行时完整性、P0-P3风险分级和最终交付报告。

## 2. 触发条件
- 用户说"质检 / 最后检查 / 看看还有没有bug / 是否可以交付"
- 修复完成后要求复检
- 打包前要求验收
- 安装包发布前要求确认
- 多轮 bug 闭环后要求避免重复问题
- 阶段冻结/基线发布前检查

## 3. 反触发条件
- 单个文件读取 → 02_wenjian_guanli
- 单个文档解析 → 03_wendang_chuli
- 单个代码问题定位 → 04_daima_zhenduan
- 只运行测试 → 13_zhongduan_zhixing
- 只生成 zip → 05_xiangmu_dabao

## 4. 输入契约
- project_root：项目根目录
- 当前版本号/包名
- 本轮质检范围（全部 / 特定模块 / 特定轮次闭环）
- 是否包含历史问题复检
- 是否允许运行测试
- 是否允许打包
- 交付标准
- 不允许混入的新增轮次问题（如"只检查第 8/9 轮闭环，不混第 10 轮"）

## 5. 输出契约
最终必须输出：
- 总结论：通过 / 部分通过 / 不通过 / 禁止发布
- P0/P1/P2/P3 问题清单
- 已验证通过项
- 未验证项（单独列出，不能藏在通过结论里）
- 风险项
- 证据路径/日志/工具输出
- 修复建议（每个未通过问题）
- 复检建议
- 是否允许打包 / 是否允许发布

## 6. 工具选择矩阵

**scan_project**
- 用途：建立项目结构和关键文件清单
- 适用：所有收口质检起点
- 失败恢复：缩小范围，转 02_wenjian_guanli

**run_python_quality_check**
- 用途：Python 语法/测试/质量检查
- 适用：后端、工具、脚本、Runtime
- 失败恢复：traceback 转 04_daima_zhenduan

**run_python_tests**
- 用途：在受控工作区执行 pytest 测试
- 适用：代码修复后回归验证、发布前质量检查
- 失败恢复：失败用例和 traceback 转代码诊断

**diagnose_project**
- 用途：综合诊断项目结构、依赖、配置、入口
- 适用：启动失败、打包失败、跨模块问题
- 失败恢复：拆分为 02_wenjian_guanli + 04_daima_zhenduan

**evaluate_quality_gate**
- 用途：执行质量门裁决
- 适用：收口结论生成前
- 失败恢复：人工规则降级裁决，输出无法自动判定原因

**build_delivery_standardization**
- 用途：生成标准化交付报告
- 适用：通过或部分通过后
- 失败恢复：列出缺失字段，不允许伪完整

## 7. 标准执行流程

**阶段 A：质检边界确认**
- 明确检查哪个版本、哪个包、哪些历史问题
- 防止混入用户明确排除的问题范围

**阶段 B：结构扫描**
- scan_project
- 确认入口文件、配置、运行时、资源、Skill、工具注册完整性

**阶段 C：静态质量检查**
- run_python_quality_check
- 前端/JS 如无专用工具，记录"需外部检查"，不伪执行

**阶段 D：综合诊断**
- diagnose_project
- 检查路径、Provider、工具链、Skill 对齐、运行时完整性

**阶段 E：关键链路 smoke**
根据项目类型逐项检查：
- 启动链路
- Provider 配置链路
- 工具调用链路
- 上传文件链路
- 工作/聊天模式路由
- 打包完整性
- 内置 Python runtime 完整性

**阶段 F：质量门裁决**
- evaluate_quality_gate
- 按 P0-P3 输出分级清单

**阶段 G：交付标准化**
- build_delivery_standardization
- 输出正式报告

**阶段 H：复检建议**
- 每个未通过问题给出修复优先级和复检命令/步骤

## 8. 风险分级规则

- P0（阻断）：阻断启动、阻断后端、数据损坏、安装包缺关键运行时、核心工具链不可用 → 禁止发布
- P1（高优）：核心功能可启动但关键任务失败，如 Provider 工具调用断、上传文件无法读、模式路由错 → 不建议发布
- P2（中优）：稳定性、体验、一致性问题，如报告不完整、入口不一致、错误提示不清 → 可内测但列已知问题
- P3（低优）：文案、样式、非阻断优化 → 不影响发布

## 9. 证据链规则
- 每个问题必须有证据（文件路径+行号+日志）
- 每个"已修复"必须有复检证据
- 每个"通过"必须说明检查范围
- 每个"未验证"必须单独列出，不得藏在通过结论里
- 不得把模型推测当作质检证据

## 10. 失败恢复
- 工具无法执行：降级静态检查，标记未验证
- 运行时缺失：P0，禁止发布
- 测试依赖缺失：标记环境限制，输出复现命令
- 历史问题无法确认：列为未验证，不得判定已修复
- 发现新问题：单独列"新增问题"，不混入指定轮次闭环

## 11. 发布裁决规则
- 存在 P0 → 禁止发布
- 存在核心 P1 → 不建议发布，只可内部验证
- 仅 P2/P3 → 可灰度或内测，必须列已知问题
- 全部关键链路通过 → 允许打包发布

## 12. 协作规则
- 文件证据 → 02_wenjian_guanli
- 文档报告 → 03_wendang_chuli
- 代码根因 → 04_daima_zhenduan
- 测试执行 → 13_zhongduan_zhixing
- 生成 zip → 05_xiangmu_dabao

## 13. 验收用例
1. 完整项目发布前质检：输出 P0-P3 清单
2. 只复检第 8/9 轮 bug，不混第 10 轮
3. 安装包缺 Python 标准库 → 判定 P0，禁止发布
4. Provider 工具调用签名不匹配 → 判定 P1
5. 上传文件只注入元数据未强制读取 → 判定 P1/P2
6. 打包成功但 manifest 缺失 → 判定 P2
7. 工具执行失败时输出未验证项清单
8. 无 P0/P1 时允许内测发布
