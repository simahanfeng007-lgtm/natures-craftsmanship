# 代码诊断

## 1. 定位
代码诊断负责识别项目中的逻辑bug、语法错误、依赖错误、路径错误、打包错误、Provider调用错误、工具链错误、测试失败和架构错位。不直接修改代码，而是走"复现取证 → 定位 → 根因分类 → 修复建议 → 验证计划"的工程路线。

## 2. 触发条件
- 用户说"检查有没有bug / 逻辑bug / 启动不了 / 报错"
- 用户上传 zip 包或代码项目要求质检
- 用户贴出 traceback、API 400、ImportError、TypeError、路径错误
- 用户要求确认修复是否完成
- 用户要求分析 Provider、工具调用、Runtime、前后端链路问题
- 终端执行返回失败后要求找根因

## 3. 反触发条件
- 只是读普通文档 → 文档处理
- 只是查文件是否存在 → 文件管理
- 只是执行命令或测试 → 终端执行
- 只是打包发布 → 项目打包
- 最终发布验收 → 项目收口质检
- 用户明确说"不要修改，只聊天"时，仅做只读分析

## 4. 输入契约
- 项目根目录或 zip 解包目录
- 用户提供的错误现象（traceback、报错信息、症状描述）
- 是否允许执行测试：是/否
- 是否允许修改代码：是/否
- 目标平台：Windows / Linux / macOS / Electron / Python
- 验收标准：启动成功、测试通过、无回归、特定bug消失

## 5. 输出契约
每次诊断最终输出：
- 问题清单（序号 + 描述）
- 严重等级：P0（阻断）/ P1（高优）/ P2（中）/ P3（低）
- 影响范围
- 证据：文件路径、函数名、行号或代码片段、错误日志
- 根因判断
- 修复建议（明确改哪个文件、哪个函数、为什么）
- 验证方法（语法检查、单元测试、冒烟测试等）
- 是否需要修改代码的声明

如果用户要求"只检查不修改"，必须只输出诊断，不得执行写入。

## 6. 工具选择矩阵

**diagnose_project**
- 用途：项目级诊断，发现结构、依赖、配置、入口问题
- 适用：上传完整项目、启动失败、打包失败
- 失败恢复：转文件管理建立目录树，缩小范围

**run_python_quality_check**
- 用途：Python 语法检查、pytest、基础质量检查
- 适用：Python 后端、脚本、工具模块
- 失败恢复：读取报错文件，定位具体语法/导入/依赖问题

**run_python_tests**
- 用途：在受控工作区执行 pytest 测试，验证代码修复或项目质量
- 适用：代码修复后的回归测试、项目质量验证
- 失败恢复：提取失败测试名和 traceback，转代码诊断定位根因

**tool_skill_blueprint**
- 用途：生成工具/Skill对齐蓝图
- 适用：工具注册、Skill引用、Runtime对齐问题
- 失败恢复：输出缺失映射和人工核查清单

**return_analysis**
- 用途：输出诊断报告（只读）
- 适用：只读分析、问题清单、修复建议

**return_code**
- 用途：输出建议代码或补丁片段
- 适用：用户要求修复方案或代码草案
- 约束：如果没有写入工具，不得声称已修改文件

**model_chat**
- 用途：必要时调用模型进行辅助分析
- 适用：复杂架构判断、多文件逻辑推理
- 约束：必须以实际文件证据为输入，不得空泛问模型

## 7. 标准执行流程

**阶段 A：复现与事实收集**
- 读取用户报错（traceback、日志）
- scan_project / diagnose_project 建立项目结构
- 如允许，运行 run_python_quality_check

**阶段 B：问题定位**
- 有错误栈 → 按错误栈定位文件和行号
- 无错误栈 → 按入口文件、配置、Provider、工具注册、前后端接口逐层排查
- 所有代码判断必须有 read_file 返回的实际文件证据

**阶段 C：根因分类**
至少区分以下类型：
- 语法错误、导入错误、依赖缺失
- 路径错误、编码错误
- Provider/API 参数错误
- 工具调用签名不匹配
- 前后端接口不一致
- 状态机/模式路由错误
- 打包缺件、逻辑条件错误
- 异常吞掉或异常分类错误

**阶段 D：修复建议**
- 优先最小变更方案
- 明确：改哪个文件、哪个函数/行号、为什么这样改
- 核心 Runtime 保持低侵入
- 用户未授权修改 → 只给建议，不写文件

**阶段 E：验证计划**
- 语法检查 → 单元测试 → 冒烟测试 → 启动测试 → 回归测试 → 真实路径测试

**阶段 F：收口**
- 输出 P0/P1/P2/P3 清单
- 标记：已确认 / 疑似需验证 / 无法验证
- 给出下一步修复顺序

## 8. 证据链规则
- 每个 bug 都必须有证据（文件路径 + 行号/片段 + 错误输出）
- 不能说"可能有问题"，必须说明依据
- 疑似问题标记"需运行验证"并给出复现步骤
- 修复建议要能回溯到证据
- 未读取代码时不得给出确定性结论

## 9. 失败恢复
- 项目无法解包 → 转文件管理或要求重新上传
- 测试无法运行 → 先做静态诊断，标记"未执行测试"
- 依赖缺失 → 不盲装依赖，输出缺失依赖和安装建议
- 报错信息不足 → 按入口链路逐层排查
- 大项目扫描超时 → 缩小到入口、配置、报错关键词
- 测试失败原因不明 → 保留日志，分层排查

## 10. 质量门
- 通过：每个问题有证据、根因明确、修复建议可验证
- 部分通过：证据不足但给出复现路径和验证计划
- 不通过：无证据猜测、跳过测试、无法定位却给确定结论

## 12. 验收用例
1. Python ImportError 诊断：定位缺失模块和导入链
2. TypeError 参数不兼容诊断：定位签名不匹配位置
3. Electron 前后端接口不一致诊断：对比接口契约
4. Provider API 400 参数错误诊断：检查请求体
5. 工具注册缺失诊断：输出缺失工具清单
6. 用户要求"只检查不修改"：只输出诊断报告
7. 用户要求"确认这个bug是否修好"：对比前后证据
8. 代码诊断后输出 P0/P1/P2/P3 清单
